## 1. 概要・目的の比較

### 1.1 目的

| 項目 | Confluence | Jira |
|------|-----------|------|
| **主な目的** | 仕様書の検索・要約・深掘り | 課題データの検索・要約・深掘り |
| **対象ユーザー** | 開発者、プロダクトマネージャー | 開発者、プロダクトマネージャー |
| **解決する課題** | 情報が分散しており、目的の仕様を探し出すのに時間がかかる | 仕様書（Confluence）と課題管理（Jira）を横断的に検索 |
| **統合の位置づけ** | 基盤システム | Confluence検索システムへの統合 |

### 1.2 主要機能

| 機能 | Confluence | Jira |
|------|-----------|------|
| **自然言語検索** | ✅ | ✅ |
| **AI要約** | ✅ | ✅ |
| **ストリーミング回答** | ✅ | ✅ |
| **会話履歴管理** | ✅ | ❌（未実装） |
| **参照元リンク表示** | ✅ | ✅ |
| **ソース切替** | ❌（単一ソース） | ✅（タブ切替） |
| **Jira特有フィールド表示** | ❌ | ✅（ステータス、優先度、担当者、Issue Key等） |

**主な違い**:
- Confluenceは会話履歴管理機能があるが、Jiraには未実装
- Jiraはソース切替機能とJira特有フィールド表示がある

## 2. システムアーキテクチャの比較

### 2.1 全体構成

| 項目 | Confluence | Jira |
|------|-----------|------|
| **UI層** | Next.js UI | Next.js UI（タブ切替） |
| **API層** | API Routes | API Routes |
| **検索エンジン** | ハイブリッド検索エンジン | ハイブリッド検索エンジン |
| **ベクトルDB** | LanceDB (`confluence`テーブル) | LanceDB (`jira_issues`テーブル) |
| **全文検索** | Lunr.js (Confluence専用インデックス) | Lunr.js (Jira専用インデックス) |
| **AI生成** | Gemini API | Gemini API |

**主な違い**:
- **テーブル分離**: ConfluenceとJiraで異なるLanceDBテーブルを使用
- **インデックス分離**: Lunrインデックスもテーブルごとに独立
- **UI**: Jiraはタブ切替UIを追加

### 2.2 データフロー

#### データ同期フロー

| ステップ | Confluence | Jira |
|---------|-----------|------|
| **1. データ取得** | Confluence APIからページデータを取得 | Jira REST API (`/rest/api/3/search/jql`) から課題データを取得 |
| **2. ページネーション** | Confluence APIの標準ページネーション | `nextPageToken`を使用（`startAt`は無視） |
| **3. データ整形** | HTMLからテキストを抽出 | 課題データを正規化・整形 |
| **4. チャンク分割** | 1800文字、オーバーラップなし | ❌（チャンク分割なし） |
| **5. ベクトル生成** | Gemini Embeddings API (768次元) | Gemini Embeddings API (768次元) |
| **6. データ保存** | LanceDBに保存 | Firestore `jiraIssues` + LanceDB `jira_issues` |
| **7. 同期履歴** | ❌ | Firestore `jiraSyncJobs`に記録 |

**主な違い**:
- **チャンク分割**: Confluenceは1800文字でチャンク分割、Jiraは分割なし
- **保存先**: ConfluenceはLanceDBのみ、JiraはFirestore + LanceDBの2段階保存
- **同期履歴**: Jiraは同期履歴をFirestoreに記録

#### 検索フロー

| ステップ | Confluence | Jira |
|---------|-----------|------|
| **1. クエリ変換** | Gemini Embeddings APIで埋め込みベクトルに変換（768次元） | 同様 |
| **2. ベクトル検索** | LanceDB `confluence`テーブルで類似ベクトル検索 | LanceDB `jira_issues`テーブルで類似ベクトル検索 |
| **3. キーワード検索** | LanceDB LIKE句によるタイトル・コンテンツ検索 | ❌ |
| **4. BM25検索** | Lunr.jsによる全文検索 | Lunr.js Jira専用インデックスによる全文検索 |
| **5. タイトル厳格一致検索** | タイトルがクエリに完全一致する検索 | ❌ |
| **6. 早期ラベルフィルタリング** | 議事録、アーカイブ等の除外 | ❌ |
| **7. 動的キーワード抽出** | ドメイン知識データベースから関連キーワードを抽出 | ❌ |
| **8. スコアリング統合** | RRF融合 + Composite Scoring | RRF融合 + Composite Scoring |
| **9. 結果補完** | ❌ | ベクトル検索結果から完全なレコードを取得（Jira特有フィールドを補完） |
| **10. BM25結果補完** | ❌ | BM25検索結果をLanceDBでenrichment（Jira特有フィールドを補完） |
| **11. 重複除去** | ✅ | ✅ |
| **12. 回答生成** | Gemini API | Gemini API |

**主な違い**:
- **検索タイプ**: Confluenceは4種類（ベクトル、キーワード、BM25、タイトル厳格一致）、Jiraは2種類（ベクトル、BM25）
- **ラベルフィルタリング**: Confluenceは早期ラベルフィルタリングあり、Jiraはなし
- **動的キーワード抽出**: Confluenceのみ
- **結果補完**: Jiraは検索後にJira特有フィールドを補完する処理がある

## 3. データスキーマの比較

### 3.1 LanceDBスキーマ

#### Confluenceスキーマ

```typescript
interface ConfluenceRecord {
  id: string;                    // チャンクID (pageId-chunkIndex)
  vector: number[];              // 768次元の埋め込みベクトル
  page_id: number;               // ConfluenceページID（int64型）
  chunkIndex: number;            // チャンク番号
  space_key: string;             // スペースキー
  title: string;                 // ページタイトル
  content: string;               // チャンク内容
  url: string;                   // ページURL
  lastUpdated: string;           // 最終更新日時（ISO文字列）
  labels: string[];              // ラベル配列（空配列も可）
  isChunked: boolean;            // チャンク分割済みフラグ
  totalChunks: number;           // 総チャンク数
}
```

#### Jiraスキーマ

```typescript
interface JiraRecord {
  id: string;                    // Issue Key (例: "CTJ-1234")
  vector: number[];              // 768次元の埋め込みベクトル
  issue_key: string;             // Issue Key (例: "CTJ-1234")
  title: string;                 // Summary（課題のタイトル）
  content: string;               // Description + コメント
  status: string;                // ステータス名（例: "完了", "進行中"）
  status_category: string;       // ステータスカテゴリ（例: "完了", "進行中"）
  priority: string;              // 優先度（例: "High", "Medium"）
  assignee: string;              // 担当者（例: "池田広大", "(unassigned)"）
  issue_type: string;            // 課題タイプ（例: "ストーリー", "バグ"）
  reporter: string;              // 報告者
  created_at: string;            // 作成日時（ISO文字列）
  updated_at: string;            // 更新日時（ISO文字列）
  labels_text: string;           // ラベル（カンマ区切り文字列）
  project_key: string;           // プロジェクトキー（例: "CTJ"）
  project_name: string;          // プロジェクト名
  url: string;                   // Jira課題URL
}
```

### 3.2 スキーマ比較表

| フィールド | Confluence | Jira | 説明 |
|-----------|-----------|------|------|
| **id** | `pageId-chunkIndex` | `Issue Key` | 一意識別子 |
| **vector** | ✅ (768次元) | ✅ (768次元) | 埋め込みベクトル |
| **page_id / issue_key** | `page_id` (number) | `issue_key` (string) | ソース固有ID |
| **chunkIndex** | ✅ | ❌ | チャンク番号（Confluenceのみ） |
| **space_key / project_key** | `space_key` | `project_key` | スペース/プロジェクトキー |
| **title** | ✅ | ✅ | タイトル |
| **content** | ✅ (チャンク内容) | ✅ (Description + コメント) | コンテンツ |
| **url** | ✅ | ✅ | ソースURL |
| **lastUpdated / updated_at** | `lastUpdated` | `updated_at` | 最終更新日時 |
| **labels** | `labels` (string[]) | `labels_text` (string) | ラベル |
| **isChunked** | ✅ | ❌ | チャンク分割フラグ（Confluenceのみ） |
| **totalChunks** | ✅ | ❌ | 総チャンク数（Confluenceのみ） |
| **status** | ❌ | ✅ | ステータス（Jiraのみ） |
| **status_category** | ❌ | ✅ | ステータスカテゴリ（Jiraのみ） |
| **priority** | ❌ | ✅ | 優先度（Jiraのみ） |
| **assignee** | ❌ | ✅ | 担当者（Jiraのみ） |
| **issue_type** | ❌ | ✅ | 課題タイプ（Jiraのみ） |
| **reporter** | ❌ | ✅ | 報告者（Jiraのみ） |
| **created_at** | ❌ | ✅ | 作成日時（Jiraのみ） |
| **project_name** | ❌ | ✅ | プロジェクト名（Jiraのみ） |

**主な違い**:
- **チャンク分割**: Confluenceはチャンク分割に対応（`chunkIndex`, `isChunked`, `totalChunks`）、Jiraは分割なし
- **ID形式**: Confluenceは`page_id`（number）、Jiraは`issue_key`（string）
- **ラベル形式**: Confluenceは配列、Jiraは文字列（カンマ区切り）
- **Jira特有フィールド**: ステータス、優先度、担当者、課題タイプなど

### 3.3 Firestoreスキーマ

| コレクション | Confluence | Jira |
|------------|-----------|------|
| **データ保存** | ❌（LanceDBのみ） | ✅ `jiraIssues` |
| **同期履歴** | ❌ | ✅ `jiraSyncJobs` |
| **会話履歴** | ✅ `conversations` | ❌（未実装） |

**主な違い**:
- ConfluenceはFirestoreにデータを保存しない（LanceDBのみ）
- JiraはFirestoreにrawデータと正規化データを保存
- Jiraは同期履歴をFirestoreに記録

## 7. データ同期の比較

### 7.1 同期方式

| 項目 | Confluence | Jira |
|------|-----------|------|
| **APIエンドポイント** | Confluence REST API | `/rest/api/3/search/jql` |
| **ページネーション** | Confluence API標準 | `nextPageToken`（`startAt`は無視） |
| **クエリ** | Space Keyベース | JQL: `project = "CTJ" ORDER BY updated DESC` |
| **取得フィールド** | ページデータ全般 | `summary`, `description`, `status`, `priority`, `assignee`, `reporter`, `created`, `updated`, `labels`, `issuetype`, `project`, `comment` + カスタムフィールド |
| **差分取得** | ✅ | ✅（既存データの`updated`フィールドと比較） |
| **LanceDB反映** | 直接保存 | 差分アップサート（テーブルが空または欠損している場合のみ全件再構築） |

### 7.2 同期スクリプト

| コマンド | Confluence | Jira |
|---------|-----------|------|
| **全データ同期** | `npm run sync:confluence:batch` | `npm run sync:jira` |
| **差分同期** | `npm run sync:confluence:differential` | ❌（差分取得機能により自動的に差分のみ更新） |
| **最大件数指定** | ❌ | `JIRA_MAX_ISSUES=200 npm run sync:jira` |

### 7.3 同期処理フロー

| ステップ | Confluence | Jira |
|---------|-----------|------|
| **1. データ取得** | Confluence APIからページデータを取得 | Jira APIから課題データを取得（`nextPageToken`を使用） |
| **2. データ整形** | HTMLからテキストを抽出 | 課題データを正規化・整形 |
| **3. 差分チェック** | ✅ | ✅（既存データの`updated`フィールドと比較） |
| **4. Firestore保存** | ❌ | ✅（`jiraIssues`、バッチ処理: 50件/バッチ、変更があったもののみ） |
| **5. チャンク分割** | 1800文字、オーバーラップなし | ❌ |
| **6. ベクトル生成** | ✅ | ✅（バッチ処理: 50件/バッチ、10並列、リトライロジック付き） |
| **7. LanceDB保存** | ✅ | ✅（差分アップサート / テーブルが空の場合は全件再構築） |
| **8. 同期履歴記録** | ❌ | ✅（Firestore `jiraSyncJobs`、追加・更新・変更なしの統計情報を含む） |

### 7.4 差分取得の詳細

| 項目 | Confluence | Jira |
|------|-----------|------|
| **比較方法** | 未記載 | 既存データの`updated`フィールドとJira APIから取得した`updated`フィールドを比較 |
| **更新判定** | 未記載 | Jiraの`updated`が既存データより1秒以上新しい場合のみ更新 |
| **統計情報** | 未記載 | `added`（追加）、`updated`（更新）、`unchanged`（変更なし）の件数 |
| **パフォーマンス** | 未記載 | 変更がないデータはスキップされるため、同期時間とAPIコール数を削減 |

## 12. まとめ

### 12.1 主な相違点

1. **データ構造**
   - Confluence: チャンク分割対応、Firestore保存なし
   - Jira: チャンク分割なし、Firestore + LanceDBの2段階保存

2. **検索機能**
   - Confluence: 4種類の検索タイプ、ラベルフィルタリング、動的キーワード抽出
   - Jira: 2種類の検索タイプ、Jira特有フィールドの補完処理

3. **UI/UX**
   - Confluence: 会話履歴管理あり
   - Jira: ソース切替UI、Jira特有フィールド表示

4. **パフォーマンス**
   - Confluence: 7-23ms（高速）
   - Jira: 100-300ms（やや遅い）

5. **運用**
   - Confluence: 手動同期中心
   - Jira: 定期同期推奨、同期履歴管理あり

### 12.2 共通点

1. **技術スタック**: Next.js、TypeScript、Gemini API、LanceDB、Lunr.js
2. **検索方式**: ハイブリッド検索（ベクトル + BM25）
3. **スコアリング**: RRF融合 + Composite Scoring
4. **ストリーミング**: SSE形式のストリーミング回答

### 12.3 今後の改善提案

1. **Jiraの検索性能向上**: 100-300ms → 7-23msを目指す
2. **Jiraの会話履歴管理**: Confluenceと同様の機能を追加
3. **Confluenceの同期履歴管理**: Jiraと同様の機能を追加
4. **統一的なエラーハンドリング**: 両システムで統一的なエラー処理を実装
5. **横断検索**: ConfluenceとJiraを同時に検索する機能
